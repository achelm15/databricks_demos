Stop Writing Custom Data Quality Frameworks: A Tour of Databricks Labs DQX

Profiles, YAML rules, Python rules, and row-level quarantine — without writing your own runner.

By Alexander Booth

Every data engineer has written a run_check() helper.

Take a SQL predicate, count the failing rows, insert a row into a results table, and print a pass/fail icon. Maybe loop over a list of rule dicts. Maybe add a severity column. Maybe even a notes field, escaped exactly once because you got burned by a single quote.

I have one for a prior project, 06_reliability_dq.ipynb (linked here). It works. It graphs. It alerts. It also does nothing that the Databricks Labs DQX library doesn’t do better.

This post is the case for throwing yours away.

The Dataset

A synthetic season of pitch-level baseball data, ~50,000 rows, generated locally with Faker and landed in Unity Catalog as dqx_baseball_bronze.pitches_raw.

Roughly 12% of rows have planted quality issues:

  • Null pitcher_id or batter_id
  • Pitch speeds that are negative or greater than 130 mph
  • Innings beyond the 20th
  • Batting averages outside [0, 1]
  • Bogus pitch type codes like XX or ??
  • Game dates in the future
  • Duplicate at_bat_ids

Every issue maps to at least one rule we’ll write, so the rule-level breakdown at the end is interpretable.

A quick smoke check on bronze confirms the planted mess:

+-----+------------+---------+----------+-------+--------+-----------+
|total|null_pitcher|bad_speed|bad_inning|bad_avg|bad_type|future_date|
+-----+------------+---------+----------+-------+--------+-----------+
|50000|641         |1295     |693       |1342   |671     |663        |
+-----+------------+---------+----------+-------+--------+-----------+

That’s the input. Now to the question of what to run against it.

What “Good Enough” Looks Like — and Why It Isn’t

Here is the helper from the hand-rolled notebook, in full:

def run_check(layer, table_name, rule_name, predicate_fail, severity="warn", notes=""):
    """predicate_fail = SQL predicate that matches FAILING rows."""
    failing = spark.sql(f"SELECT COUNT(*) c FROM {table_name} WHERE {predicate_fail}").collect()[0]["c"]
    total   = spark.sql(f"SELECT COUNT(*) c FROM {table_name}").collect()[0]["c"]
    passed  = failing == 0
    spark.sql(f"""
        INSERT INTO {dq_table} VALUES (
            '{RUN_ID}', TIMESTAMP'{RUN_TIME}', '{layer}', '{table_name}',
            '{rule_name}', '{severity}', {failing}, {total}, {str(passed).lower()},
            '{notes.replace("'", "''")}'
        )
    """)
    icon = "PASS" if passed else ("FAIL" if severity == "error" else "WARN")
    print(f"  [{icon}] {rule_name}: {failing}/{total} failing")
    return passed

It is fine. It is also a series of shortcuts.

The notes.replace("'", "''") line is a SQL injection patch that proves the point… we are interpolating strings into SQL because we never wired up parameterized queries. The semantics are inverted (the predicate matches failing rows, not passing ones), which is easy to get wrong at 5 pm on a Friday. Uniqueness has to be re-implemented from scratch:

run_check("silver", f"{S}.game_data g", "silver_game_sk_unique",
          f"game_sk IN (SELECT game_sk FROM {S}.game_data GROUP BY game_sk HAVING COUNT(*) > 1)",
          severity="error", notes="PK uniqueness")

The dq_results table was designed by hand. The severity column is a free-form string. There is no rule linter, so a typo goes unnoticed and the rule passes.

And the deepest problem: this approach only ever counts failures. It never tells you which rows failed. There is no quarantine. If a downstream consumer wants the bad rows isolated for review — and they always do — you build that subsystem yourself.

DQX is the subsystem.

What DQX Actually Is

DQX is a pure-Python library from Databricks Labs. The engine accepts checks as either YAML metadata or Python objects, builds Spark plans that execute server-side, and returns DataFrames annotated with _errors and _warnings columns.

It has three primitives:

  • DQProfiler — scans a DataFrame, returns summary statistics plus per-column profiles (nulls, ranges, distinct values).
  • DQGenerator — turns those profiles into candidate rules.
  • DQEngine — validates, applies, and splits.

Because it builds Spark plans rather than executing locally, DQX works fine through Databricks Connect. Everything in this post runs serverless from a local Jupyter notebook.

A note on install. If your network blocks PyPI (some corporate setups have 127.0.0.1 pypi.org in /etc/hosts), pull the wheels straight from GitHub releases:

gh release download v0.14.0 -R databrickslabs/dqx       --pattern "*.whl" -D /tmp --clobber
gh release download v0.12.0 -R databrickslabs/blueprint --pattern "*.whl" -D /tmp --clobber

pip install --no-deps /tmp/databricks_labs_dqx-0.14.0-py3-none-any.whl \
                      /tmp/databricks_labs_blueprint-0.12.0-py3-none-any.whl

The databricks-sdk~=0.73 pin in DQX’s wheel metadata is tighter than the actual requirement — DQX only needs WorkspaceClient, which works fine on older SDK versions. --no-deps sidesteps the pin.

Profile → Rules in Five Lines

This is the most “magic” part of DQX and the part least covered online.

Point DQProfiler at the bronze table, hand the result to DQGenerator, and you get a candidate ruleset:

from databricks.sdk import WorkspaceClient
from databricks.labs.dqx.profiler.profiler import DQProfiler
from databricks.labs.dqx.profiler.generator import DQGenerator

ws = WorkspaceClient()
df = spark.read.table(BRONZE)

profiler   = DQProfiler(ws)
summary, profiles = profiler.profile(df)

generator   = DQGenerator(ws)
auto_checks = generator.generate_dq_rules(profiles, criticality="warn")

For the pitch table that produces 13 candidate rules — one or two per column, covering not-null and range/in-list constraints. A sample of the output:

- check:
    function: is_not_null_and_not_empty
    arguments:
      column: at_bat_id
      trim_strings: true
  name: at_bat_id_is_null_or_empty
  criticality: warn

- check:
    function: is_in_range
    arguments:
      column: game_date
      min_limit: '2026-03-28'
      max_limit: '2026-07-02'
  name: game_date_not_in_range
  criticality: warn

There are two important things to notice.

First, the pitch_speed_mph range candidate pulls its bounds from the polluted data. The profiler sees the −72 mph rows and the 140 mph rows and dutifully proposes a range that contains them. The DQX docs are explicit about this: auto-generated rules are candidates, not production rules. A hand-curated checks.yml uses 40–110.

Second, profiling is a one-time discovery step. You do not schedule it. You run it, review the output, tighten the thresholds, and commit a final YAML.

YAML and Python — Same Engine, Two Styles

The hand-curated checks.yml for this dataset is seven rules:

- criticality: error
  check:
    function: is_not_null_and_not_empty
    for_each_column:
      - at_bat_id
      - game_id
      - pitcher_id
      - batter_id

- name: pitch_speed_in_range
  criticality: error
  check:
    function: is_in_range
    arguments:
      column: pitch_speed_mph
      min_limit: 40
      max_limit: 110

- name: pitch_type_in_enum
  criticality: warn
  check:
    function: is_in_list
    arguments:
      column: pitch_type
      allowed: [FF, SI, SL, CU, CH, FC, KC, FS]

- criticality: error
  check:
    function: sql_expression
    arguments:
      expression: game_date <= current_date()
      msg: game_date is in the future

- criticality: error
  check:
    function: is_unique
    arguments:
      columns: [at_bat_id]

YAML covers the 80% case. For things YAML can’t express cleanly — dynamic rules built from another system, or rules that need runtime config — you reach for Python:

from databricks.labs.dqx.rule import DQRowRule, Criticality
from databricks.labs.dqx import check_funcs

py_checks = [
    DQRowRule(
        name="pitcher_id_format",
        column="pitcher_id",
        check_func=check_funcs.regex_match,
        check_func_kwargs={"regex": r"^P\d{4}$"},
        criticality=Criticality.WARN.value,
    ),
    DQRowRule(
        name="speed_above_min",
        column="pitch_speed_mph",
        check_func=check_funcs.sql_expression,
        check_func_kwargs={
            "expression": "pitch_speed_mph IS NOT NULL AND pitch_speed_mph > 0",
            "msg": "pitch_speed_mph must be positive",
        },
        criticality=Criticality.ERROR.value,
    ),
]

The two styles use the same engine. Same _errors and _warnings semantics. You can mix them in a single pipeline.

Before running anything, DQX lints the YAML:

yaml_status = DQEngine.validate_checks(yaml_checks)
print(f"YAML checks  ({len(yaml_checks)} rules): {yaml_status}")
# YAML checks  (7 rules): No errors found

Typo a function name — is_in_rage instead of is_in_range — and the lint fails loudly before a single Spark task runs. The hand-rolled notebook has no equivalent step. A typo there silently produces a passing rule.

Apply, Split, Quarantine

This is where the comparison gets brutal.

from databricks.labs.dqx.engine import DQEngine

dq_engine = DQEngine(ws)
df = spark.read.table(BRONZE)

# Pass 1: YAML rules
valid_yaml_df, invalid_yaml_df = dq_engine.apply_checks_by_metadata_and_split(df, yaml_checks)

# Pass 2: Python rules on the survivors
valid_clean = valid_yaml_df.drop("_errors", "_warnings")
valid_df, invalid_py_df = dq_engine.apply_checks_and_split(valid_clean, py_checks)

# Final invalid set
invalid_df = invalid_yaml_df.unionByName(invalid_py_df, allowMissingColumns=True)

The result:

Input rows: 50,000
Valid:      45,291
Invalid:    6,070   (11.8%)

invalid_df does not just have a count. It has the original row plus an _errors array (rules with criticality error) and a _warnings array (rules with criticality warn). Each array element is a struct containing the rule name, the failure message, the columns involved, and the function that fired.

Land the valid rows in silver, the invalid rows in quarantine, and you have row-level audit trails for free:

valid_df.drop("_errors", "_warnings") \
    .write.mode("overwrite").option("overwriteSchema", "true").saveAsTable(SILVER)

invalid_df.write.mode("overwrite").option("overwriteSchema", "true").saveAsTable(QUARANTINE)

(INSERT OVERWRITE rather than MERGE because Databricks Connect doesn’t support MERGE. Not a DQX limitation.)

The hand-rolled notebook has nothing comparable. It counts; DQX tags, splits, and lands.

A Sanity Check from Quarantine

One query against the quarantine table gives the rule-level breakdown:

WITH unnested AS (
    SELECT explode(_errors) AS e FROM dqx_baseball_quarantine.pitches_quarantine
    UNION ALL
    SELECT explode(_warnings) FROM dqx_baseball_quarantine.pitches_quarantine
)
SELECT e.name AS rule, COUNT(*) AS failures
FROM unnested
GROUP BY e.name
ORDER BY failures DESC

+---------------------------+--------+
|rule                       |failures|
+---------------------------+--------+
|batting_avg_not_in_range   |1342    |
|pitch_speed_in_range       |1295    |
|inning_not_in_range        |693     |
|batter_id_is_null_or_empty |677     |
|pitch_type_in_enum         |671     |
|not_game_date_current_date |663     |
|pitcher_id_is_null_or_empty|641     |
|at_bat_id_is_not_unique    |100     |
+---------------------------+--------+

Lines up almost perfectly with what we planted in bronze. The differences (677 vs 641 for batter ID nulls, 100 for at_bat_id duplicates) are random sampling on the mutator side, not DQX missing anything.

The Longitudinal Table

Roll the quarantine breakdown into a long-format Delta table, keyed by (run_id, run_time, rule_name):

INSERT INTO dq_results
WITH bronze_total AS (SELECT COUNT(*) AS total FROM bronze_table),
failures AS (
    SELECT e.name AS rule_name, 'error' AS severity
    FROM quarantine LATERAL VIEW explode(_errors) AS e
    UNION ALL
    SELECT w.name, 'warn'
    FROM quarantine LATERAL VIEW explode(_warnings) AS w
),
rolled AS (
    SELECT rule_name, severity, COUNT(*) AS failing_rows
    FROM failures
    GROUP BY rule_name, severity
)
SELECT '<run_id>' AS run_id,
       TIMESTAMP'<run_time>' AS run_time,
       'bronze_table' AS table_name,
       r.rule_name, r.severity, r.failing_rows,
       bt.total AS total_rows,
       1.0 - (r.failing_rows / bt.total) AS pass_rate
FROM rolled r CROSS JOIN bronze_total bt

This is the exact shape the hand-rolled dq_results table already has. That matters. If you are already alerting and dashboarding off your home-grown table, DQX changes how you populate it, not what’s downstream. The upgrade is local.

Run the pipeline on a schedule, and after ten executions you can plot pass-rate-by-rule against time. The orange line is pitch_speed_in_range regressing through the SLA threshold. That is the kind of signal you get for free once row-level data quality is structured and longitudinal — and the kind of signal that is hard to surface when your DQ pipeline is a stack of run_check() calls.

Where DQX Still Has Rough Edges

A short list, since I’ve been honest about everything else.

  • DBR floor. DQX supports DBR 14.3 LTS and up. Older runtimes need not apply.
  • The wheel-metadata pin. As mentioned, databricks-sdk~=0.73 is tighter than the actual requirement. pip install --no-deps works around it.
  • Profiler sample defaults. DQProfiler.profile() samples 30% of rows up to a 1,000-row cap by default. Fine for discovery, not enough for production. Tune via opts={...}.
  • No MERGE through Databricks Connect. Use INSERT OVERWRITE or saveAsTable(mode="overwrite") instead. Not DQX’s fault, but worth flagging.
  • Two-pass for mixed checks. Running YAML rules and Python rules in sequence requires dropping the result columns between passes (apply_checks_* rejects DataFrames that already have _errors/_warnings). Easy once you know — the kind of thing that costs you half an hour the first time.

Bottom Line

The case isn’t that DQX is magic. It’s that DQX is the framework you’d write yourself — written better, maintained by people who think about data quality for a living, and shipped with the parts you usually skip because they’re tedious.

Three concrete wins over the hand-rolled approach:

  • Row-level quarantine — the part you almost never build yourself, and the part downstream consumers always want.
  • A rule linter — validate_checks catches the typo before Spark does.
  • A profiler — five lines to a starting ruleset, with the candidates ready for review.

If you have a run_check() helper sitting in your repo, the upgrade path is short. Same dq_results shape downstream. Same dashboards. Throw out the helper, keep the table.
